# un9flow 开发哲学

> **Deterministic Embedded Intelligence**

## 一、核心命题

`un9flow` 的核心命题只有一句话：

> **在安全关键嵌入式系统中，确定性不是实现风格，而是系统生存条件。**

这意味着项目的第一优先级不是“抽象是否优雅”，而是：

- 资源是否可预测
- 时序是否可解释
- 故障是否可定位
- 异常是否可收敛到安全态

在电力电子、BMS、驱动控制和功能安全场景中，失败不是普通的软件缺陷，而可能直接转化为器件损伤、误动作或系统级风险。因此，`un9flow` 以**确定性、可观测性、可验证性**作为设计前提。

## 二、三大约束公理

### 1. un-dynamic → 静态资源公理

- 默认禁止 `malloc` / `free`
- 内存布局必须在编译期或链接期清晰定义
- 关键变量、缓冲区、保留区、掉电保持区必须可追踪
- 显式 section、Scatter 文件、Linker Script 是架构的一部分，而不是实现细节

核心思想：

> **关键资源必须先被定义，再被使用。**

### 2. un-scheduled → 时间确定性公理

- 默认不依赖不可解释的动态调度行为
- 优先超级循环、确定节拍轮询或严格约束的调度模型
- ISR 只承担必要、短路径、可终止的职责
- 超时、看门狗喂养、复位与降级路径必须前置设计

核心思想：

> **所有关键时序都必须可测量、可解释、可证明。**

### 3. un-linked → 单向收敛公理

- 数据流优先单向传播，避免相互等待与双向耦合
- 状态机必须有默认安全态、超时退路和异常收敛路径
- 通信、控制与保护逻辑必须避免形成不可控闭环

核心思想：

> **系统在异常条件下必须收敛，而不是纠缠。**

## 三、四阶段工作流

### Phase 1: Hazard Analysis

目标：先建立风险边界，再讨论实现路径。

关注点：

- 最坏后果是什么
- 哪些非功能约束高于功能本身
- 默认安全态是什么
- 哪些失效模式最不可接受

标准输出：

- 生存清单
- 失效模式清单
- 非功能约束基线
- Failsafe 定义

### Phase 2: Deterministic Foundation

目标：把资源、状态、时序和寄存器行为固化为可验证结构。

关注点：

- 寄存器位域是否明确
- 内存布局是否静态且可审计
- ISR 与主循环职责是否清晰
- 状态机是否具备默认安全收敛路径

标准输出：

- 寄存器位域地图
- 静态内存布局
- ISR / 主循环职责表
- 状态机收敛说明

### Phase 3: Link Diagnostics

目标：让链路、采样、隔离与通信路径具备足够可观测性。

关注点：

- 是否能定位到具体掉线段、具体器件、具体 hop
- 是否提前定义观测点、时序窗口和诊断顺序
- 是否具备最小诊断闭环，而不是依赖经验猜测

标准输出：

- 链路健康诊断表
- 分段排查步骤
- 观测点与波形清单
- 异常定位策略

### Phase 4: Failsafe Validation

目标：通过故障注入和极限工况验证系统的安全收敛能力。

关注点：

- 欠压、过压、过温、掉线、CRC 错误、ADC 漂移、时序抖动下会发生什么
- 系统能否进入 Limp Home 或 Failsafe
- 看门狗、保护链和复位策略是否按预期动作

标准输出：

- 故障注入矩阵
- 高风险失效路径清单
- 跛行模式验证表
- 安全态收敛结论

## 四、系统观

`un9flow` 对嵌入式系统的理想状态有四个要求：

### 1. Deterministic
资源、状态、时序都必须稳定可预测。

### 2. Observable
关键变量、关键链路、关键波形必须可观测。

### 3. Explainable
任何关键行为都必须能追溯到明确的寄存器、状态机或时序条件。

### 4. Failsafe
任何异常都必须优先导向安全收敛，而不是继续冒险运行。

## 五、九重约束模型

`un9flow` 将以下九类因素视为不可压缩约束：

1. 电压
2. 温度
3. 时序
4. CRC / 数据完整性
5. 死区时间
6. 栈深
7. ADC 精度与漂移
8. 隔离耐压
9. 看门狗与复位链

这九类约束共同构成系统工程边界。只要其中任一项缺乏控制，系统就不应被视为“已完成”。

## 六、一句话总结

> **un9flow = 以确定性为第一原则，为安全关键嵌入式系统建立可验证、可诊断、可收敛的工程方法。**
