# fault-injection-report-example

## case overview
### 案例缺口清单
- v6 当前还没有一份明确偏“故障注入结果收口”的示例案例，导致收口路径只能从分散文档中拼接理解。
- 仓库虽然已有 incident / watchdog / findings / report / generator 等能力与文档，但缺少一份把“注入了什么、观察到什么、如何收口”串成单案例闭环的报告型样例。
- 需要先用范围可控的注入场景，验证 evidence → specialist → findings → report 这条收口链条在实操中可用。

本案例选择“末端节点通信间歇失联”作为单点注入场景，目标是给出一份可复用的收口报告骨架，而不是构建完整试验数据库或长期测试计划。

## injection setup
- 场景对象：8 节点串行通信链路中的末端节点 Node-8。
- 注入方式：在 Node-8 与上游节点之间周期性插入通信静默窗口（每 4 秒一次，静默 180 ms，持续 25 分钟）。
- 控制边界：仅注入链路间歇失联，不改动业务控制策略与保护阈值。
- 采集范围：总线时序日志、watchdog 触发记录、重试计数、incident 时间线。
- 最小复现实验入口（骨架）：通过 fault-injector 注入入口执行 `--target Node-8 --period 4s --silence 180ms --duration 25m`，从 Node-8 上游链路侧注入，并记录本次执行批次 ID。
- 时间基准对齐：以注入命令生效时刻记为 T0，evidence / watchdog / incident 全部换算为 `Δt = timestamp - T0` 后再比对。
- 观察窗口：`[T0-2min, T0+28min]`（含注入前基线 2 分钟、注入期 25 分钟、注入后恢复观察 3 分钟）。
- 判定门槛：恢复=静默窗口结束后 ≤2 个轮询周期恢复且后续 3 个注入周期无连续 timeout；未恢复=连续 ≥3 个注入周期在窗口结束后仍 timeout；需要升级=10 分钟内出现 ≥2 次“未恢复”或任一系统级保护触发。

### evidence-package 摘要
- 时间戳序列显示 Node-8 在注入窗口内出现连续心跳缺失，窗口外恢复正常应答。
- 上游节点重试计数与失联窗口高度对齐，非注入时段无异常尖峰。
- watchdog 记录到多次短时 timeout 事件，但未触发系统级保护停机。
- incident 时间线可将“注入时刻—失联观测—恢复时刻”三段对齐到同一时间轴。

## observed behavior
- 注入开始后第一个 5 分钟（约 75 个注入周期）内，Node-8 间歇失联在绝大多数周期（约 >90%）都贴近静默窗口出现；窗口结束后多数周期在 ≤2 个轮询周期内恢复。
- 全观察窗口内仅见少量恢复延迟（约 <10% 周期），表现为单次窗口结束后仍需 1~2 个轮询周期才恢复。
- 同一观察窗口内，其余 7 个节点未出现连续 ≥2 次 timeout，未观察到与 Node-8 同步比例的丢包放大。
- 业务层局部数据新鲜度下降主要集中在注入期，且未出现连续 >30 秒的全链路阻塞。

## incident interpretation
- 该事件更符合“链路层间歇可达性下降”而非“节点硬故障离线”。
- 已有证据支持注入与 timeout 事件之间存在时间相关性，但仍不能仅凭本案例确认唯一根因。
- 现阶段可解释为：注入造成末端节点短时不可达，触发 watchdog timeout 记录与上游重试堆积，随后在链路恢复后逐步收敛。

### risk-boundary-note 摘要
- 已确认：注入窗口会稳定诱发末端节点间歇失联观测，并能被 evidence 与 watchdog 记录捕获。
- 未确认：是否存在底层物理层噪声、终端匹配或驱动抖动等放大因素共同参与。
- 边界声明：本案例用于验证收口链条可用性，不用于给出最终硬件根因裁决。

## watchdog / report synthesis
本案例按“evidence → specialist → findings → report”推进：
1. evidence-package 汇总可复现的原始观测；
2. watchdog timeout specialist 对 timeout 分布与重试堆积进行解释；
3. 形成 findings，再由 report 汇总可确认结论与边界；
4. 输出面向评审的收口结果。

### watchdog-timeout-audit-findings 摘要
- timeout 事件在时间上集中于注入窗口附近，分布与注入节律一致。
- 事件等级以短时可恢复为主，未形成持续不可恢复态。
- 建议将末端节点 timeout 统计与重试退避参数联动评估，以减少恢复抖动。

### watchdog-timeout-audit-report 摘要
- 报告结论：该注入案例成功证明了从观测证据到审计报告的收口流程可执行。
- 报告边界：当前报告确认“可复现现象与可解释链条”，不声称“唯一根因已锁定”。
- 报告动作：保留后续专项验证项，并将本案例作为 v6 故障注入报告模板基线。

## final outcome
### 已确认项
- 已构建一个范围可控的“末端节点通信间歇失联”注入案例。
- 已跑通 evidence → specialist → findings → report 的收口链条。
- 已形成包含结论边界的报告型案例文本，可用于后续同类案例复用。

### 未确认项
- 末端节点恢复延迟的底层触发因素（协议栈节拍、物理层扰动、驱动实现差异）尚未区分。
- 在更长时长与更高注入频率下，是否会触发更高等级故障尚无证据。

### 下一步建议
- 在本案例同参数下补做一次复跑，并按 `Δt` 时间轴对齐复核恢复判定门槛是否稳定。
- 将本案例已使用的“恢复/未恢复/需要升级”判定与证据字段固化为检查清单项。
- 对已出现的少量恢复延迟追加一次短时回放核对（不扩大注入范围），排除采集抖动干扰。
