# BMS Daisy-Chain AFE End-to-End Example

## case overview

### 案例缺口清单
- v6 当前没有真正意义上的端到端方法链示例：缺少一份从首次 bring-up 到风险边界收敛的连续样例。
- 仓库虽然已有 bring-up / incident / watchdog / design-safety-review 对象，但还没有说明“何时切场景、何时切专项、每一步产出什么”的完整串联文档。
- 需要先用一个足够真实、足够小的 BMS 场景把方法链跑通：以“菊花链 AFE 首次拉通后节点枚举不稳定，伴随偶发 timeout / reset 风险”为主线。

### 场景定义
- 系统：8 节点菊花链 AFE，首次板级联调。
- 现象：上电后链路可通，但节点枚举数量在 6~8 间抖动；长稳压测中偶发通信 timeout，个别样机出现自动 reset。
- 目标：不是一次性“彻底修复”，而是形成可复用的方法链切换样例，明确每一步输入、输出和边界。

## bring-up path usage

### 何时留在 bring-up path
满足以下条件时继续走 bring-up：
- 现象仍以“首次拉通不稳定”为主，问题可通过基础时序、链路配置、供电与复位窗口检查解释。
- 还未形成可复现故障窗口，不需要事件分级和跨角色协同。

### bring-up 阶段动作
1. 固定上电与采样窗口，减少测试过程噪声。
2. 记录每次枚举结果与链路质量指标（CRC 错误计数、重试次数、节点响应延迟）。
3. 输出最小可复现实验序列（摘要骨架），作为后续 incident 输入：
   - 前置条件：固定供电、温度与负载窗口，锁定样机版本与固件版本。
   - 操作顺序：按统一上电序列连续执行多轮（如 20 轮）节点枚举与基础采样。
   - 观测点：每轮记录枚举节点数、CRC/重试计数、节点响应延迟、是否出现 timeout/reset。
   - 判定条件：同一条件下若仍出现枚举抖动或 timeout（如 ≥3/20 轮），判定为“可重复不稳定”，触发 incident handoff。

### bringup-baseline 摘要
- 输入：初版硬件样机 + 默认链路参数 + 标准上电序列。
- 关键记录：100 次冷启动中，枚举稳定 61 次，缺失节点主要集中在末端 2 节点。
- 直接结论：问题不是“完全不可通信”，而是“可通但稳定性不足”。
- 输出去向：提供 incident 初始事实基线。

### link-qualification-log 摘要
- 记录范围：链路驱动强度档位、时钟分频、帧间隔、重试策略。
- 观察结果：提高帧间隔后，枚举抖动有所收敛；但在高温与低压角落条件下仍出现 timeout。
- 阶段判定：已超出单纯 bring-up 参数微调可快速闭环的范围，进入 incident handoff。

## incident handoff

### 何时从 bring-up 切到 incident
- 在固定供电/温度/负载条件下按统一序列重复若干轮（如 20 轮）后，仍出现可重复但非必现的不稳定（如枚举抖动或 timeout ≥3 轮）。
- 故障已影响系统关键路径（节点识别和采样连续性），且需要统一事件编号、时间线、影响面描述，避免各自记录口径不一致。

### incident 阶段产出
- 建立事件条目：包含触发条件、影响范围、当前缓解措施。
- 给出临时运行约束：例如限制在特定温度/电压窗口内运行。
- 明确下一跳：将“timeout / reset 风险”拆出，转 watchdog 专项分析。

## watchdog / timeout specialty

### 何时切入 watchdog 专项
- incident 已确认 timeout 与 reset 已进入关键路径，且仅靠链路参数调整已无法稳定解释现象。
- 需要判断是通信栈阻塞触发 watchdog，还是电源/复位链误触发导致看门狗误判。

### watchdog-timeout-audit-findings 摘要
- 已确认现象：timeout 前存在响应延迟拉长与重试堆积。
- 未确认机制：reset 是 watchdog 正常保护动作，还是异步复位噪声触发。
- 关键证据缺口：缺少 reset 前后 200 ms 的统一高分辨率时序对齐数据。

### watchdog-timeout-audit-report 摘要
- 风险分级：中到高（在长稳场景会破坏采样连续性）。
- 临时建议：
  - 保守放宽超时阈值并增加告警分层，先降低误复位概率。
  - 增加事件埋点：超时入口、喂狗点、复位源寄存器快照。
- 交接说明：涉及保护策略边界，需进入 design safety review 进行“可接受风险”判定。

## design safety review handoff

### 何时从 watchdog 切到 design safety review
- 技术上可提出临时缓解，但该缓解会触动保护灵敏度或风险边界（如放宽阈值/延后动作窗口）。
- 仅靠专项内参数调整无法同时满足“降低误触发”与“不提升漏检风险”，需要系统级权衡。

### risk-boundary-note 摘要
- 当前可接受边界（暂定）：在受控工况下允许低概率重试增加，但不接受无告警 reset。
- 明确不可接受项：任何导致节点丢失后静默运行的路径。
- 评审决议（阶段性）：在补齐复位源证据前，不批准长期放宽 watchdog；仅允许带监控的临时参数。

## final outcome

### 已确认项
- 菊花链 AFE 的问题核心是“枚举稳定性不足 + 偶发 timeout”，并已形成跨场景统一事实基线。
- bring-up 到 incident 的切换条件明确：当问题进入可重复但非必现且影响关键路径时必须升级。
- watchdog 专项已识别关键证据缺口，并输出了可执行的临时防护措施。

### 未确认项
- reset 的根因链条尚未闭合：watchdog 触发与复位链噪声的因果关系仍待证实。
- 高温低压角落条件下的末端节点失联是否与硬件批次差异相关，尚无充分统计结论。

### 下一步建议
1. 追加带统一时钟基准的 200 ms 前后窗口采样，优先补齐 reset 根因证据。
2. 在不放宽长期安全边界前提下，继续使用“分层告警 + 临时参数”策略收集数据。
3. 若两轮样机后仍存在同型 timeout/reset，升级为硬件-软件联合设计复盘。

---

该案例用于展示方法链切换与交接，不代表问题已完美解决，也不替代完整 BMS 设计白皮书。
